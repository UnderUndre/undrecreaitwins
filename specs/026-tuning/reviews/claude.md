## Summary

The cross-artifact consistency is genuinely good — FR↔task coverage is complete, the dependency graph is valid, the clarifications (2026-06-22/23) resolved real ambiguities, and the deferral of Method B is clean. **But the plan rests on an integration surface that does not exist.** Three of the load-bearing "reuse" APIs (`LLMClient` JSON mode, `ValidatorPipeline.runDryRun`, `FunnelRepository.createFunnelVersion`) are named in research.md with 90–100% reuse figures and are **not present in the codebase** — the real surfaces have different names, different signatures, and in one case a different conceptual model (validators check *replies*, not *prompts*). On top of that, the headline atomicity guarantee of the P1 activate path (FR-006 "all in a single DB transaction") is **unachievable** with the reused repositories, because each one opens and commits its own `withTenantContext` transaction. The internal `analyze.md` returned PASS 0/0 — expected, because it audited artifact consistency, not whether the cited engine APIs are real. This is exactly the blind spot Principle VI exists to catch.

Net: the spec is well-structured but not implementation-ready. An agent following research.md literally will write code against phantom APIs, fail, and improvise — the precise AI-codegen failure mode this repo's own guardrails warn about.

## Findings

| ID | Severity | Area | Finding | Recommendation |
|---|---|---|---|---|
| F1 | CRITICAL | Data integrity / Transactions | FR-006 / T014 ([tasks.md:71](specs/026-tuning/tasks.md)) require activate to be atomic: "save snapshot → supersede → update persona → create funnel version → set validator toggles → mark activated. **All in a single DB transaction**." This is impossible with the reused repos. `withTenantContext` ([db.ts:18-26](packages/core/src/db.ts)) wraps each call in its **own** `db.transaction`; `PersonaRepository.update` ([persona-repository.ts:100-133](packages/core/src/services/persona-repository.ts)) and `FunnelRepository.createVersion` ([funnel-repository.ts:153-211](packages/core/src/services/funnel/funnel-repository.ts), which opens its *own* top-level `db.transaction`) each **commit independently**. If funnel creation throws after the persona update commits, live config is left **partially activated with no atomic rollback** — corrupting the active persona/funnel. The US2 acceptance ("rollback → persona reverted") cannot cover this failure path. | Either (a) refactor the touched repos to accept an injected `tx` and run the whole activate inside one `withTenantContext` transaction, or (b) explicitly downgrade FR-006 to "best-effort with compensating rollback on partial failure" and specify the compensation. Do not claim single-transaction atomicity that the data layer can't deliver. |
| F2 | HIGH | LLM / Extraction core | `LLMClient.complete()` sends only `model, messages, temperature, max_tokens` ([llm-client.ts:65-70](packages/core/src/services/llm-client.ts)); the `LLMRequest` interface ([llm-client.ts:14-20](packages/core/src/services/llm-client.ts)) has **no `response_format` field**. research.md §1.6 claims it "supports `response_format: { type: 'json_object' }`" and spec.md rates LLM client "100% reuse". It does not. The entire doc-extraction pipeline (T008/T013, data-model §5 `ExtractionOutput`) and the interview draft-gen (T020) depend on enforced JSON output that isn't wired. Without JSON mode, free-form models return prose, making the "unparseable JSON" fallback (T025) the *common* path, not the edge case. | Add an explicit task to extend `LLMClient.complete()` (+`LLMRequest`) to pass `response_format` through to the provider body. Re-rate LLM client reuse as "extend, not reuse". Keep the regex fallback but treat it as defense-in-depth, not the primary parser. |
| F3 | HIGH | Quality gate / Validators | `ValidatorPipeline.runDryRun(systemPrompt, messages)` (research.md §1.5, "90% reuse"; FR-013; T012) **does not exist**. The real pipeline ([validators/pipeline.ts:32-132](packages/core/src/services/validators/pipeline.ts)) exposes `validateResponse(reply, context)` and `validateInput(input, context)`; "dry-run" is a *per-validator mode* read from `validator_configs` ([pipeline.ts:95,98](packages/core/src/services/validators/pipeline.ts)), not a callable. More fundamentally, validators judge a **generated reply**, not a **systemPrompt** — so FR-013 "run validator dry-run on the generated systemPrompt" is a category error. And `validateResponse` requires a `conversationId` and persists to `validator_runs` ([pipeline.ts:214-237](packages/core/src/services/validators/pipeline.ts)); at generation time there is no conversation. SC-004 ("block-rate <30%") has **no defined input corpus** — you cannot compute a block-rate without sample replies. | Redefine FR-013/SC-004: to gate quality you must (1) generate N sample replies from the draft prompt against a fixed probe-message set, then (2) run each through `validateResponse` with a synthetic conversation context, then (3) aggregate block-rate. Specify the probe set and the synthetic-context shape, or cut the quality gate from v1. |
| F4 | HIGH | Activate / Funnel mapping | `FunnelRepository.createFunnelVersion(personaId, funnelConfig)` (research.md §1.4 "100% reuse"; T014) **does not exist**. The real API is `createVersion(definitionId, config, stages[], slots[])` ([funnel-repository.ts:153](packages/core/src/services/funnel/funnel-repository.ts)) and requires a pre-existing `funnelDefinition` (via `createFunnel`/`getActiveVersion`, [funnel-repository.ts:43,144](packages/core/src/services/funnel/funnel-repository.ts)). The draft's flat `funnelConfig` JSONB and the LLM's `funnelStages: [{name, description, triggers, slots[]}]` (data-model §5) must be **decomposed** into `stages` (each with `fragments` — which the LLM output does not produce) + `slots` arrays, and a definition must be resolved/created first. This mapping layer is unspecified and is not "pass the config directly". | Add a task for a `funnelConfig → {definition, stages[], fragments[], slots[]}` mapper, define where `fragments` come from (LLM? defaulted?), and correct the API name/signature in research.md §1.4 and T014. This is on the P1 critical path — it blocks the tracer-bullet. |
| F5 | HIGH | Security / Tenant isolation | The data-model RLS policy (§1.3) deviates from every sibling table and is weaker on the write side. Established pattern: `ENABLE` **+ `FORCE ROW LEVEL SECURITY`**, 2-arg `current_setting('app.current_tenant', true)`, and `WITH CHECK` on write-heavy tables ([0007_hermes_foundation.sql:59-64](drizzle/0007_hermes_foundation.sql), [0009_per_assistant_mcp.sql:75-88](drizzle/0009_per_assistant_mcp.sql), [0002_fresh_shockwave.sql:42-47](drizzle/0002_fresh_shockwave.sql)). The 026 policy has **no `FORCE`** (table-owner role bypasses RLS), **no `WITH CHECK`** (INSERT/UPDATE not tenant-constrained → cross-tenant writes possible — and `tuning_drafts` is write-heavy: status transitions, activate, reaper), and uses **single-arg `current_setting('app.current_tenant')`** which *throws* `unrecognized configuration parameter` when the GUC is unset (any non-tenant connection) instead of returning NULL. | Mirror the sibling pattern exactly: `ENABLE` + `FORCE ROW LEVEL SECURITY`, `USING (...) WITH CHECK (...)`, and 2-arg `current_setting('app.current_tenant', true)`. (`tenant_id text` itself is fine — it matches `personas.tenant_id`, [personas.ts:26](packages/core/src/models/personas.ts).) |
| F6 | HIGH | Async / RLS / Crash-safety | The fire-and-forget pattern (research.md §7, T009: `process.nextTick(() => runGenerationPipeline(draftId, tenantId))`) has two unspecified hazards. (1) **No tenant context**: the request transaction is gone after the 202, so the background pipeline's writes to the RLS-forced `tuning_drafts` (generating→ready/failed) **must** be wrapped in `withTenantContext(tenantId, …)` or RLS blocks them — the precedent exists ([llm-retry-worker.ts:119](packages/core/src/services/llm-retry-worker.ts)) but the spec never states it, and research §7's snippet omits it. (2) **Unhandled rejection**: `process.nextTick(() => asyncFn())` attaches no `.catch`; any throw (e.g. `ServiceUnavailableError` at [llm-client.ts:75](packages/core/src/services/llm-client.ts), a DB error) becomes an unhandledRejection → process-level crash risk. The timeout→failed edge case assumes a clean catch the wiring doesn't guarantee. | Spec the pipeline wrapper: `process.nextTick(() => runGenerationPipeline(...).catch(err => markDraftFailed(draftId, tenantId, err)))`, and mandate `withTenantContext` for every background DB write. State the tenant-context requirement in T008. |
| F7 | HIGH | Liveness / Deadlock | FR-011 ("only one `generating` draft per persona") + the reaper running **only on poll** (FR-003) creates a permanent deadlock. If the client stops polling (tab closed, crash) a stalled draft stays `generating` forever — nothing reaps it — so **every future generate for that persona returns 409 `CONFLICT_DRAFT_ACTIVE` indefinitely**. There is no TTL, no background sweep, no escape hatch. | Add a time-based escape: either (a) treat a `generating` draft older than the stall threshold as non-blocking in the FR-011 lock check, or (b) run the reaper opportunistically inside `generate` (sweep this persona's stale drafts before the lock check), or (c) a periodic sweep. Without one of these, one orphaned draft bricks tuning for a persona. |
| F8 | HIGH | Rollback correctness | FR-007 / T016 restore `previousSnapshot` to "live **persona** config", but activate (FR-006) also creates a new **funnel version** (deactivating the prior one) and sets **validator toggles**. Rollback restores none of the latter two, so after rollback the prompt reverts while the funnel stays on the draft's new version and validator toggles stay changed → **partial, inconsistent restore** on the P1 path. `previousSnapshot` is untyped `jsonb` (data-model §1.1) with no defined scope. | Define `previousSnapshot` to capture persona config **+ prior funnel active-version id + prior validator toggles**, and make rollback reverse all three (reactivate the prior funnel version, restore toggles). Or constrain v1 activate to persona-prompt-only and defer funnel/validator activation — but then FR-006 steps 4–5 must move out of v1. |
| F9 | MEDIUM | Concurrency | T026 claims the FR-011 lock is "race-condition safe via DB-level check", but data-model §1.2 defines only **non-unique** indexes. An app-level `SELECT generating? then INSERT` is a TOCTOU race — two concurrent generates both see no generating draft and both insert. | Add a partial unique index: `CREATE UNIQUE INDEX ... ON tuning_drafts (persona_id) WHERE status = 'generating';` and handle the unique-violation as the 409. That is the only actually race-safe enforcement. |
| F10 | MEDIUM | Edge case / API | `method` enum includes `template-bootstrap` (data-model §1.4, FR-001) but Method B is deferred with zero pipeline. T009 only "validates the method enum" — so `template-bootstrap` passes validation, fires the background task, hits no handler, and the draft sits `generating` until reaped to `failed` after 90s (or triggers the F7 deadlock). Silent, confusing failure. | Hard-reject `template-bootstrap` at the generate route with 400/501 `METHOD_NOT_IMPLEMENTED` until the follow-up spec lands. Add an edge-case task + test. |
| F11 | MEDIUM | Performance / SLA | The three time budgets are not reconciled against the *sequential* pipeline. SC-001 = 60s draft; LLM timeout = 55s (research §2.3); reaper = 90s (FR-003). After a legitimate 55s extraction, the quality-gate dry-run (F3, itself LLM-bound and unbudgeted) + funnel mapping + writes run, easily exceeding the 60s SC-001 — yet the 90s reaper still considers the draft healthy, so it never flags an SC-001 violation. | Budget the full critical path (extract + gate + map + write), set the reaper threshold below or aligned with the SLA intent, and give the quality gate its own latency budget — or run it asynchronously after marking the draft `ready`. |
| F12 | MEDIUM | Architecture / Alternatives | T018 says "patch `ChatService.buildSystemPrompt()` to inject a draft overlay". That method is **private** with signature `(tenantId, persona, userMessage, funnelCtx)` ([chat-service.ts:1108](packages/core/src/services/chat-service.ts)), called at [:373](packages/core/src/services/chat-service.ts) and [:804](packages/core/src/services/chat-service.ts). Overlaying `funnelConfig` and `validatorToggles` (not just the prompt) also requires the funnel runtime and `ValidatorPipeline` (which both load tenant/persona config from the DB) to accept overrides — a deeper change than the "80% reuse" framing. A cleaner alternative was not weighed: construct a **shadow `persona`/funnel-context object** with overridden fields and pass it through the existing public entrypoint, avoiding private-method surgery. | Evaluate the shadow-config approach vs. patching internals; pick one explicitly in research.md. Re-scope the sandbox tasks to include funnel + validator override plumbing, not just the prompt. |
| F13 | MEDIUM | State / Ambiguity | Interview state is keyed `tuning:interview:{tenantId}:{personaId}` (data-model §2.2) — a **single concurrent interview per persona per tenant**; two admins tuning the same persona silently clobber each other's session. Separately, `next` vs `answer` responsibility for advancing `currentQuestion` is ambiguous: US3 scenarios imply `answer` records and `next` serves the current unanswered question, but T020 has `next` "advance to next question" — calling `next` twice could skip a question. | Key the session per interviewer (include a session/user id), and define precisely which endpoint advances the cursor. Add a test for double-`next` and for concurrent interviewers. |
| F14 | LOW | Spec drift | FR-001 enumerates the `tuning_drafts` columns but omits `error`, while data-model §1.1/§1.4 includes `error text` and the reaper/timeout paths write `error: 'GENERATION_STALLED' | 'LLM_TIMEOUT'`. (The internal analyze missed this.) | Add `error` to the FR-001 column list so the requirement and schema agree. |
| F15 | LOW | Cache consistency | Proposals are stored under two Redis keys with the same TTL — the set `tuning:proposals:{tenant}:{persona}` and per-item `tuning:proposal:{proposalId}` (data-model §3.2). `reject` (T023) dismisses the individual key but not the entry in the set, so a rejected proposal reappears on the next `GET /proposals` until the set TTL lapses. The two keys can also expire independently. | On `reject`, also remove the item from the cached set (or store proposals in one structure). Document the single source of truth between the two keys. |
| F16 | LOW | AuthZ (verify) | FR-003 says the poll endpoint has "no auth beyond tenant scoping (tenantId from header)". The contract requires `Bearer` + `X-Tenant-ID`, and RLS is defense-in-depth, but the phrasing invites a header-trust bug: a caller must not be able to read another tenant's drafts by setting `X-Tenant-ID` to a tenant their token doesn't own. | Confirm the shared auth hook binds the Bearer token to the allowed tenant(s) and rejects mismatched `X-Tenant-ID` before RLS — and state that the tuning routes inherit it. Flagged for verification, not confirmed as a hole. |

## Alternative approaches considered

- **Generation execution model (F1/F6):** in-process fire-and-forget is a reasonable v1 call given the existing BullMQ workers were deemed heavyweight — but the spec already depends on BullMQ for document parse/chunk/embed (plan.md §Primary Dependencies). Reusing the existing worker infra for generation would solve F6 (tenant context + crash-safety are already handled by `llm-retry-worker`) and F7 (a queue gives you visibility/retry instead of a poll-only reaper) at the cost of latency. Worth a one-paragraph justification in research.md for why in-process beats the queue you already run, rather than asserting it.
- **Sandbox overlay (F12):** shadow-persona object vs. patching private `ChatService` internals — the former is less invasive and avoids threading an overlay param through two public call sites.
- **Quality gate (F3):** if a defined probe-corpus is too much for v1, an alternative is to drop the automated block-rate gate and surface `confidence` from the LLM's self-report only (data-model §5 already returns `confidence`), deferring validator-based gating to a follow-up.
