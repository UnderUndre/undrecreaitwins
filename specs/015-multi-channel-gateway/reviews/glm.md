# SpecKit Review: 015-multi-channel-gateway

**Reviewer**: glm
**Reviewed at**: 2026-06-09T03:15:00+03:00

## Summary

Solid spec that correctly reuses existing twin-engine infrastructure (ChannelAdapter, Redis Streams, validators 004) rather than reinventing. The two gate-0 prerequisites (CL-A6 reengagement bypass, CL-A1 plaintext creds) are properly identified and sequenced. Main weaknesses: (1) cross-repo spec/impl mismatch violates Principle IX and isn't just a flag — it's an active risk for drift between spec tree and code; (2) webhook signature verification outsourced to per-adapter implementations with no shared crypto module, creating inconsistency risk across 4+ webhook channels; (3) no provisioning API contract — tasks build adapters but nothing specifies how a tenant admin provisions a channel with encrypted creds; (4) consumer-process sprawl (15+ processes) has no resource management or coalescing strategy.

## Alternative approaches considered

1. **OUTBOUND interceptor (stopgap for CL-A6)**: instead of waiting for the twin-engine reengagement chip, add a middleware in `channel-orchestrator.ts` that intercepts all OUTBOUND writes and routes through `validateResponse()`. This solves the immediate safety issue but adds latency to every response. Trade-off: safety vs. latency. Rejected for v1 because the chip is in progress, but worth keeping as a fallback if the chip stalls.

2. **Shared webhook-gateway microservice**: instead of per-adapter webhook verification, deploy a shared HTTP gateway that receives all webhooks, verifies signatures, normalizes to `ChannelMessage`, and publishes to `twin.stream.in`. Reduces per-adapter code but adds a single point of failure and a new deployment unit. Rejected: the existing adapter-per-process pattern is cleaner and the shared crypto module (F3) addresses the consistency risk more simply.

3. **Adapter coalescing via worker threads**: instead of 15+ processes, use Node.js `worker_threads` to run multiple adapters in one process with isolated contexts. Lower memory overhead (~150 MB total vs ~750 MB). Trade-off: one adapter crash can take down co-tenants; harder to debug. Worth prototyping after MVP.
