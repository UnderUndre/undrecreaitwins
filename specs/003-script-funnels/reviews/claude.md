# SpecKit Review: 003-script-funnels

**Reviewer**: claude

## Summary

The spec is well-structured with clear requirements, measurable success criteria, and good edge-case coverage. The plan correctly mirrors the existing persona ingestion pattern. However, there are meaningful gaps in versioning strategy (funnel versions are conceptual but not modeled), authN/AuthZ on the API contract is missing, data-model drift between spec and schema is present, and several concurrency/consistency edge cases are underspecified. The spec is solid for MVP but needs tightening before implementation.

## Findings

| ID | Severity | Area | Finding | Recommendation |
|---|---|---|---|---|
| F1 | HIGH | Logical consistency | **Funnel versioning modeled as a single row, not a version chain.** Spec FR-016 requires pinning in-flight conversations to their starting version and new conversations adopting the latest. Data model has a single `funnels` row with `version` (BIGINT) and `is_active` (Boolean). Publishing v2 would require either mutating the same row (breaking pinned conversations) or creating a new row — but the schema has no `parent_funnel_id` or `version_number` column to distinguish versions. The `version` column is described as "Optimistic locking" in data-model.md, conflating concurrency control with semantic versioning. | Add a `version_number` column (separate from optimistic-lock `version`) OR model funnel versions as separate rows linked by a `funnel_definition_id` (stable identity across versions). Clarify in data-model.md which approach is used. |
| F2 | HIGH | Security | **API contract has zero AuthN/AuthZ.** `contracts/funnel-api.yaml` has no `security` section, no `401`/`403` responses, and no mention of tenant-scoping. Quickstart examples pass `X-Tenant-ID` as a plain header with no auth token. The codebase already has `tenantPlugin` + `authPlugin` — the contract should reflect this. | Add `security` section to funnel-api.yaml referencing Bearer token + tenant header. Add 401/403 responses. Match the existing pattern from `packages/api/src/routes/personas.ts`. |
| F3 | HIGH | Edge case | **"Funnel re-published mid-turn" edge case has no task-level implementation.** Spec edge case (spec.md line 103) says "the turn uses a single consistent definition snapshot." No task addresses snapshot isolation — T007 (runtime) and T022 (version pinning) don't specify how to guarantee atomic reads during concurrent publishes. | Add explicit snapshot mechanism to T007 or T022: e.g., load funnel definition once at the start of `processMessage()` and pass it through, or use PG `SELECT ... ## Alternative Approaches Considered

1. **Pre-computed stem cache instead of runtime stemming.** Rather than stemming every incoming message + all trigger phrases at match time, pre-compute and store stemmed triggers at ingestion time (in `triggers` JSONB alongside raw phrases). This would shift the stem cost from the hot path (per-message) to ingestion (one-time). Worth weighing: the 1-5ms stem budget is already well within 100ms, so this is an optimization for very high QPS, not a current necessity.

2. **Redis-backed conversation funnel state instead of PostgreSQL.** Research §3 mentions this option but doesn't decide. For high-frequency reads/writes per conversation turn, Redis would reduce PG load. However, the existing pattern uses PG for all conversation state, and adding Redis state would complicate consistency guarantees (especially the optimistic-lock version check). Current PG approach is simpler and consistent with codebase norms.

3. **Deterministic scoring via embedding similarity instead of stem+keyword matching.** The spec explicitly requires no LLM at match time, but pre-computed embeddings (generated at ingestion) compared via cosine similarity would be a more robust matching approach than keyword/stem scoring. This was likely considered and rejected in the legacy system (embedding generation is heavier, and the legacy approach works). Flagging for awareness — not recommending a change.
