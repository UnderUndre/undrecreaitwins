# SpecKit Review: 003-script-funnels

**Reviewer**: antigravity
**Reviewed at**: 2026-05-29T13:25:00+03:00
**Commit**: c7406b404a9f8ff4f2645885338b081e8fb03f43
**Artifacts reviewed**: spec.md, plan.md, tasks.md, data-model.md, contracts/funnel-api.yaml, research.md, quickstart.md

## Summary

The spec demonstrates strong domain understanding: clear priority ordering of user stories, explicit FR → task tracing, well-defined edge cases, and a principled separation between hot-path deterministic matching and off-path async slot verification. The plan correctly mirrors existing codebase patterns (persona ingestion, Drizzle, Fastify + Zod). However, the **versioning data model conflates optimistic locking with semantic funnel versions** — a structural flaw that will cascade into runtime, pinning, and ingestion tasks. Several concurrency scenarios (concurrent turn processing, funnel deletion mid-flight, slot CAS retry exhaustion) are stated as requirements but have no concrete implementation strategy, leaving the implementer to improvise critical safety guarantees. The API contract is security-bare and incomplete relative to the spec's own requirements.

## Findings

| ID | Severity | Area | Finding | Recommendation |
|---|---|---|---|---|
| F1 | HIGH | Data model / Logical consistency | **Funnel `version` column serves double duty (optimistic lock + semantic version) but spec requires them to be independent.** `funnels.version` (BIGINT, DEFAULT 0) is described as "Optimistic locking" in data-model.md. FR-016 requires pinning conversations to a *semantic* funnel version — meaning the system must retain the full funnel definition (stages, fragments, slots, config) at the version the conversation started on. A single `version` + `is_active` flag on a mutable row cannot satisfy this: updating the row for v2 destroys v1, breaking pinned conversations. The `conversation_funnel_states.funnel_id` FK points to a single `funnels` row, not a version-specific snapshot. | Model funnel versions as **immutable snapshots**: either (a) create a `funnel_versions` table with a stable `funnel_definition_id` (identity) + `version_number` (monotonic) + full `config`/children, or (b) treat each `POST /v1/funnels` as creating a new row with the same `definition_id` but incremented `version_number`, and never mutate. The `conversation_funnel_states.funnel_id` then points to the specific version row. This is the single most important design fix before implementation. |
| F2 | HIGH | Security | **API contract (`funnel-api.yaml`) has zero security definition.** No `securitySchemes`, no `security` block, no `401`/`403` responses. Quickstart passes `X-Tenant-ID` as an unauthenticated header — any caller can impersonate any tenant. The existing codebase uses `tenantPlugin` + `authPlugin` with Bearer tokens. The contract should reflect this or it will be implemented without auth. FR-015 (tenant isolation) is unenforceable without AuthN. | Add `securitySchemes` (BearerAuth) and `security` requirement to every operation. Add `401 Unauthorized` and `403 Forbidden` responses. Add `X-Tenant-ID` as a required header parameter with validation. Match the `personas.ts` pattern exactly. |
| F3 | HIGH | Concurrency / Failure mode | **Optimistic lock retry strategy on `conversation_funnel_states` is unspecified.** FR-012 mandates CAS (compare-and-set) for slot updates. FR-006/FR-007 use the same `version` column for stage transitions. The spec says "version checking" but never defines: (a) how many CAS retries before giving up, (b) what happens on exhaustion (drop the update? queue it? error the turn?), (c) whether stage transition and slot update can race on the same `version` column — if a slot update bumps `version`, a concurrent stage transition will fail and vice versa. Under moderate concurrency (multiple tabs, webhook retries), this is a real collision surface. | Define CAS retry policy explicitly: max retries (e.g., 3), backoff strategy (immediate retry is fine for PG), and exhaustion behavior (log + skip for slots, retry-or-fail for stage transitions). Consider whether stage transitions and slot updates should use **separate version counters** or a single one with documented contention acceptance. Add this to research.md §3 and reference from T018. |
| F4 | HIGH | Edge case / Logical consistency | **"Stage objective resolved" is the core transition trigger (FR-007) but has no definition anywhere.** The spec says "advance when the current stage's objective is resolved." The data model has `funnel_stages.objective` (Text) — a human-readable string. Nothing in spec, plan, research, or tasks defines the *machine-readable* resolution criteria. Is it: (a) a specific fragment was selected (implicit: the "closing" fragment), (b) a slot was filled, (c) a keyword/pattern was matched, (d) author-defined triggers in stage config? T013 says "Implement stage transition logic (advance/regression)" with zero design. The implementer will invent the semantics, which defeats the spec's purpose. | Add a `resolution_criteria` field to `funnel_stages` (data-model.md) and `StageDefinition` (API contract). Define at least two resolution types: `fragment_selected` (specific fragment ID triggers advance) and `slot_filled` (specific slot name). Document the decision in research.md. This is a spec gap, not an implementation detail. |
| F5 | MEDIUM | Security / Input validation | **`triggers` JSONB is operator-controlled input with no size/depth limits in the API contract.** `FragmentDefinition.triggers.phrases` accepts unbounded arrays of unbounded strings. An operator could submit 10,000 phrases per fragment, each 100KB, creating a scoring DoS (stemming + comparison against all triggers per message). `triggers.synonyms` is `additionalProperties` with no key/value count limits. | Add to `funnel-api.yaml`: `maxItems` on `phrases` (e.g., 50), `maxLength` on each phrase (e.g., 500 chars), `maxProperties` on `synonyms` (e.g., 100 groups), and `maxItems` on each synonym array (e.g., 20). Enforce in Zod schema (T021). |
| F6 | MEDIUM | Failure mode | **Funnel deletion while conversations are in-flight is unaddressed.** Research.md §2 lists `DELETE /v1/funnels/:id`. FR-016 covers new version publishing. But no FR, task, or edge case covers what happens when a funnel is *deleted entirely* (not re-versioned). The `conversation_funnel_states.funnel_id` FK would reference a deleted row — either FK violation on read or orphaned state. | Choose one: (a) soft-delete only (add `deleted_at` timestamp, runtime treats deleted funnel as "no funnel" for new turns but state survives), (b) block deletion while any `conversation_funnel_states` references the funnel, (c) cascade-reset all conversations on delete. Document as FR-023 and add a task. |
| F7 | MEDIUM | Performance / Scalability | **No caching strategy for funnel definitions on the hot path.** Every `processMessage()` call hits PG to load the funnel + stages + fragments for the conversation's pinned version. At 100+ concurrent conversations per persona, this is redundant reads for the same immutable definition. Research.md lists Redis as a dependency but only discusses it for conversation state and async events, not for definition caching. The <100ms budget is tight if PG round-trip is 5-10ms per query × 3 joins. | Add a read-through cache for funnel definitions keyed by `(funnel_id, version)`. Since pinned versions are immutable, the cache never needs invalidation. Redis or in-process LRU both work. Document the caching decision in research.md and reference from T007. |
| F8 | MEDIUM | Logical consistency / Data model | **`funnel_slots` are per-funnel but slot capture is per-conversation-per-stage — no stage-scoping for slots.** `funnel_slots.funnel_id → funnels.id` but no `stage_id`. This means all slots are global to the funnel. If two stages need the same slot type with different extraction prompts, there's no way to differentiate. Also, `conversation_funnel_states.captured_slots` is a flat JSONB — no indication of *when* or *in which stage* a slot was captured, which the operator might need for analytics or re-verification logic. | Consider adding optional `stage_id` to `funnel_slots` (NULL = global to funnel). Add `captured_at` timestamp to each slot entry in `captured_slots` JSONB. Not blocking, but improves operator visibility. |
| F9 | MEDIUM | Missing specification | **`stuckAction: exit_stage` references an "author-designated exit stage" but no field for it exists.** FR-009 defines `exit_stage` as a stuck action: "transition to an author-designated exit stage." The `funnel_stages` table has `stuck_action` (per-stage override, only in data-model, not in spec — already flagged by claude.md F6). But there is no `exit_stage_id` field anywhere — the author cannot designate *which* stage to exit to. | Add `exit_stage_id` (UUID, FK → `funnel_stages`, nullable) to `funnel_stages` or to funnel `config`. Required when `stuckAction = exit_stage`. Add validation in T020/T021. |
| F10 | MEDIUM | Hidden assumption | **Scoring weight defaults ("ported from legacy") are never documented.** FR-022 and Assumptions reference "legacy defaults" for scoring weights, current-stage boost, next-stage bonus, relevance threshold, stuck threshold. None of these values appear anywhere in the artifacts. The implementer cannot verify SC-003 (95% behavioral parity) without knowing the legacy weights. Even the API contract only specifies `relevance_threshold: 0.5` and `stuck_threshold: 3` — no weights. | Document all legacy default values in research.md or a dedicated section: exact-match weight, stemmed-match weight, synonym weight, stage-boost magnitude, next-stage bonus, objection-type boost. | F11 | MEDIUM | Logical consistency | **`config.stuckAction` missing from API contract.** FR-009 and FR-022 both define `stuckAction` as a configurable per-funnel setting with 3 values (`yield_generation`, `handoff`, `exit_stage`). The API contract's `FunnelDefinition.config` only has `relevance_threshold`, `off_script_behavior`, and `stuck_threshold`. `stuckAction` is absent — the operator has no way to set it via the ingestion API. | Add `stuck_action: { type: string, enum: [yield_generation, handoff, exit_stage], default: yield_generation }` to `FunnelDefinition.config` in `funnel-api.yaml`. |
| F12 | MEDIUM | Logical consistency | **API contract missing scoring weight configuration.** FR-022 says "scoring weights, current-stage boost, next-stage bonus" MUST be configurable per funnel/assistant. The `FunnelDefinition.config` schema has no properties for these. The operator can configure threshold and behavior but not the core scoring parameters that FR-022 explicitly requires. | Add `scoring_weights`, `stage_boost`, `next_stage_bonus` (or a structured `scoring` sub-object) to `FunnelDefinition.config` in the API contract. |
| F13 | MEDIUM | Missing edge case | **Concurrent turn processing for the same conversation is unaddressed.** Two messages arriving near-simultaneously for the same conversation (e.g., user double-sends, or webhook retry + real message) will both read the same `conversation_funnel_states`, both compute a fragment match, and both try to persist updated state. The optimistic lock catches the write collision — but the second message gets a stale match result (it scored against the pre-first-turn state). No spec requirement, edge case, or task addresses turn serialization. | Add an edge case and design note: either (a) serialize turns per conversation (e.g., per-conversation Redis lock or PG advisory lock), or (b) document that under concurrent sends, the second turn may use stale state and the CAS retry will resolve it eventually. Option (a) is safer for deterministic behavior per SC-002. |

Principle V (no decorative files) and VII (snapshot tagging) are governance concerns that should be acknowledged. | Add checkboxes: "Principle V: file count proportional to feature scope" and "Principle VII: each pipeline stage tagged via snapshot-stage script." |

## Alternative Approaches Considered

1. **Immutable funnel snapshots stored as a single JSONB document per version.** Instead of normalizing into `funnels` + `stages` + `fragments` + `slots` tables, store each published version as a single JSONB blob (the full ingested definition). The scorer reads one row, one JSONB document — no joins. Mutations only happen at publish time (insert a new version row). This trades query flexibility for simplicity and performance on the hot path. The normalized model is better for querying and future editor features, but the hot path only needs a read-only snapshot. **Hybrid**: normalize for CRUD, materialize a JSONB snapshot column on publish for the scorer to read.

2. **Pre-stemmed trigger index at ingestion time.** Rather than stemming trigger phrases at match time (per-message), stem and index all triggers when the funnel is ingested. Store stemmed forms alongside raw phrases in `triggers` JSONB. At match time, only stem the user message (~1-5ms) and compare against pre-stemmed triggers. This eliminates the fragment-count-dependent stem cost on the hot path. For a funnel with 200 fragments × 10 phrases = 2000 stems, this shifts ~10ms of work from every message to a one-time ingestion cost.

3. **Per-conversation Redis lock for turn serialization (F13).** Instead of relying solely on optimistic locking (which detects but doesn't prevent concurrent state corruption), acquire a short-lived Redis lock keyed by `conversation_id` at the start of `processMessage()`. This serializes turns for the same conversation, guaranteeing deterministic behavior (SC-002). The lock TTL should match the <100ms hot-path budget plus a safety margin. This is standard for conversational systems with state.
