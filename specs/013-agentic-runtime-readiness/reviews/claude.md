# SpecKit Review: 013-agentic-runtime-readiness

**Reviewer**: claude
**Reviewed at**: 2026-06-07T22:45:00+03:00
**Commit**: 9ba3c0c01acaec9317ef0636291684b2570c7545
**Artifacts reviewed**: spec.md, plan.md, tasks.md, data-model.md, research.md, quickstart.md, contracts/hermes-runtime-preflight.contract.md, contracts/honcho-v3-client.contract.md, checklists/requirements.md, reviews/analyze.md

## Summary

Strong, surgical spec — two genuinely independent tails, clear fail-loud invariants, excellent grounding against the actual codebase (verified hermes-adapter.ts ACP handshake, honcho-client.ts legacy paths, hermes-executor.ts error handling). The main weakness is **hidden coupling in the preflight → healthy readiness path**: the spec assumes engine health checks exist and are structured enough to gate on preflight failure, but neither the spec nor the plan verifies this assumption against the actual `buildServer()` / Fastify health route implementation. Additionally, the Honcho v3 identity mapping has a subtle namespace collision risk that the research flagged at ~0.95 confidence but the tasks don't explicitly test for.

| F1 | HIGH | Hidden assumption | **Preflight gates readiness, but no existing readiness endpoint is verified.** FR-003 requires the engine to "refuse ready/healthy" on preflight failure. The plan (T006) says "wire preflight into engine boot/readiness in `packages/api` (`buildServer()`/bootstrap); failure → `AppError(...,'configuration_error')`, refuse ready/healthy." However, neither spec nor plan confirms what the current health/readiness mechanism looks like — is there a `/v1/health` route? Is it a Fastify plugin? Does it check dependency health? The quickstart says `curl -fsS http://localhost:8090/v1/health # expect healthy`, so something exists, but the plan doesn't describe how to wire the preflight result into it. If health is a static 200 today, T006 needs to create a dynamic health gate, which is more work than "wire." | Before T006, verify the current `/v1/health` implementation. If static, add a sub-task to make it dynamic (check preflight result + honcho permanent-mismatch flag). The contract (AC4) says "agents-disabled deploy → preflight skipped, engine starts normally" — this conditional gating also needs to be wired into the health route, not just the preflight function. |

This is the single biggest unknown that could invalidate the entire US2 plan. Add explicit acceptance: "T008 verifies workspace creation with tenantId as workspace_id, not just as a metadata label." |

The preflight gives a false sense of safety — it only guarantees Hermes existed at boot time, not that it will exist for every spawn. The spec doesn't discuss this gap. | Acknowledge in plan: preflight is a boot-time gate, not a per-spawn guarantee. The existing `spawn ENOENT` error path in hermes-adapter.ts (lines 259-279) already handles per-spawn failures with typed `AppError` + fallback. The preflight's job is to catch the common case (missing install), not to guarantee immutability. This is acceptable — just make it explicit so no one assumes preflight = per-turn guarantee. |

If Honcho v3.0.9 self-host defaults to auth ON with a generated token, the client will silently fail all calls (caught as `transient` → degraded). The operator would see degraded memory with no clear indication that auth is the issue. The permanent/transient classifier won't help here — a 401/403 could be either "auth required but not configured" (permanent) or "token expired" (transient). | T010 should explicitly test: (1) honcho v3.0.9 with no auth config → verify client works without `HONCHO_API_KEY`, (2) honcho with auth enabled → verify client works with `HONCHO_API_KEY`. | F6 | MEDIUM | Hidden assumption | **`hermes acp --check` output format is assumed parseable for `protocolVersion`.** The preflight contract (hermes-runtime-preflight.contract.md §Procedure step 4) says "Assert the reported ACP `protocolVersion` is compatible." If `--check` just prints "OK" and exits 0, there's no `protocolVersion` to parse. | Verify `hermes acp --check` output format in T005. If it doesn't emit protocolVersion, the preflight should fall back to: (1) run `hermes acp --check` for binary existence + basic health, (2) do a minimal ACP `initialize` handshake and read the response's `protocolVersion` field. ## Alternative approaches considered

1. **Sidecar Hermes container** (F10) — decouples Python from engine image, but breaks ACP stdio contract. Too much scope creep for this feature. Natural evolution if maintenance becomes painful.

2. **Honcho client as a separate package** — instead of rewriting `honcho-client.ts` in place, extract it into `packages/honcho-client/` with its own tests and versioning. This would isolate the v3 migration and make it reusable. Rejected for this feature: the plan's "surgical edit" approach is correct given the client has exactly one consumer (`hermes-executor`). If a second consumer appears later, extract then.

3. **Preflight as a separate CLI command** — `npx twin-engine preflight` that operators run before `docker compose up`. This would allow preflight without starting the engine. Rejected: the plan's approach (preflight at engine boot) is better because it's automatic and can't be skipped. But documenting a manual preflight command in the quickstart would be a nice addition for debugging.
