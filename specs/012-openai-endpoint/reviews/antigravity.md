## Alternative approaches considered

**1. HMAC-based key validation vs slow-hash**: The spec assumes traditional password-hashing (PBKDF2/scrypt) for API keys. An alternative: use HMAC-SHA256 with a server-side secret to derive the stored hash. This gives O(1) constant-time lookup (hash incoming key → direct DB match), no per-key salt management, and sub-millisecond validation — at the cost of the server secret becoming a single point of compromise. Many API-key systems (Stripe, AWS) use this pattern. The spec author should weigh this against the slow-hash approach, especially given the <10ms performance goal.

**2. Conversation threading via OpenAI-compatible `user` field vs custom header**: For thread dedup (FR-010), the spec could leverage the OpenAI `user` field (already part of the chat completions spec) as a session identifier, combined with a custom `X-Thread-Id` header for explicit threading. Alternatively, derive thread identity from a hash of the first N messages. Each has different UX and correctness tradeoffs that aren't explored.

**3. Separate Fastify instance for public routes**: Instead of adding public-key auth hooks to the existing server (which already has tenant-resolution + bearer hooks), spin up a separate Fastify instance (or use Fastify's `register` with prefix isolation) for the public OpenAI surface. This cleanly separates the auth concerns and avoids hook-ordering bugs. The plan doesn't consider this.
