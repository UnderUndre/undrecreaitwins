# Feature Specification: Engine Agentic Memory v2 — Continuous Learning & Personalized Context

**Feature Branch**: `022-engine-agentic-memory`
**Created**: 2026-06-18
**Status**: DRAFT
**Repo**: `undrecreaitwins` (Engine)
**Input**: `ai-agent-research-report.ru.md` (Mem0/Letta comparison), `019-feedback-loop-closure` (Base feedback system).

## 1. Описание

Текущая система фидбека (019) позволяет оператору исправлять бота в песочнице, но этот фидбек не превращается в "живой опыт" ассистента автоматически. Агент должен обладать памятью, которая накапливает факты о пользователе и уроки из прошлых ошибок в реальном времени.

022 трансформирует пассивный фидбек в **Agentic Memory Layer** (аналог Mem0), который:
1.  Извлекает долгосрочные факты (Long-term facts) о пользователе.
2.  Инжектирует релевантный опыт (Feedback snippets) в каждый ход.
3.  Обеспечивает непрерывное обучение (Continuous Learning) без переобучения весов модели.

## 2. Границы

**Это**: Логика извлечения и хранения фактов в БД (Drizzle/Postgres). Интеграция памяти в `chat-service.ts`. Пассивное обучение (система учится на лету).

**Это НЕ**: Внешние SaaS-платформы (Mem0 Cloud). Глобальное переобучение моделей.

## 3. Пользовательские истории

### US-1 — "Ты же уже это говорил" (Cross-session memory)
Клиент общался с ботом вчера и сказал, что его зовут Иван и он из Тулы. Сегодня он начинает новый чат. Бот сразу приветствует его: "Привет, Иван! Как там погода в Туле?".
**AC**: Факты о пользователе сохраняются в `user_memories` и подтягиваются при инициализации сессии.

### US-2 — Обучение на ошибках (Actionable Feedback)
Оператор в песочнице пометил ответ бота как "слишком формальный". Система извлекла урок: "Будь проще с этим клиентом". В следующем диалоге бот автоматически меняет тон.
**AC**: Фидбек из 019 векторизуется и подтягивается через семантический поиск по `feedback_memories`.

### US-3 — Саморедактирование памяти (Letta Pattern)
В процессе диалога модель видит, что юзер поменял данные (например, новый телефон). Модель вызывает инструмент `update_memory`, актуализируя профиль юзера в БД.

## 4. Функциональные требования

- **FR-001**: Таблица `agent_memories` (id, agentId, userId, category, content, embedding).
- **FR-002**: Пассивный экстрактор фактов. После каждого хода BullMQ воркер анализирует диалог и сохраняет новые факты (имя, предпочтения, боли).
- **FR-003**: Семантический поиск по памяти (pgvector). Перед генерацией ответа ассистент делает поиск топ-3 релевантных фактов и топ-2 фидбек-советов.
- **FR-004**: Управление конфликтами: если новый факт противоречит старому, старый помечается как `archived`.

## 5. Нефункциональные требования
- **NFR-1 (Privacy)**: Память строго изолирована по `tenantId`. Утечка фактов между клиентами — критическая уязвимость.
- **NFR-2 (Accuracy)**: Embedding модель для памяти (например, BGE-M3) должна поддерживать русский язык с точностью > 0.9.

## 6. Success Criteria
- **SC-001**: Бот помнит детали из диалога недельной давности.
- **SC-002**: Исправление, внесенное оператором, применяется ботом в течение 30 секунд (после асинхронной обработки).
