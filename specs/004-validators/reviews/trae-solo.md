## Summary

Substantial remediation since the first review pass. plan.md now exists with a full per-principle constitution check and four explicit design decisions. The deploy-day footgun (identity-guard active-by-default with hardcoded "Анна") is resolved via FR-015 revision + DD-003 + T017 migration. The contract type system is fixed (no more `[key: string]: any`). RLS, fail-isolation, empty-output guard, and validator execution ordering are all specified and tasked. That said, there remain meaningful gaps: the shared LLM client extraction lacks an interface contract for Phase 2 batching, the false-promise prefilter pattern list is promised but not enumerated, the T019 streaming-bypass telemetry has a plan↔tasks scope tension, and several minor cross-document drifts persist. Overall: the spec has matured well — the remaining issues are MEDIUM or below.

## Alternative Approaches Considered

1. **Pre-generation prompt hardening instead of post-hoc judge (reiterated from claude review).** The spec now mentions "port-the-moat" strategy (re-implement proven legacy guards). Prompt hardening (system prompt injection of "do not promise discounts") is complementary, not a replacement. Worth noting: Phase 1 could add a prompt-hardening instruction to the persona's system prompt as a zero-cost primary defense, with the post-gen judge as a backstop. This was not evaluated in the spec.

2. **Validator runs as an event stream (Kafka/Redis Streams) instead of PG table.** For high-volume deployments, `validator_runs` is a hot insert table. An event-stream approach would decouple audit logging from the chat path entirely. Trade-off: adds operational complexity (Kafka/Redis Streams), loses SQL queryability for operators, and the current PG approach with `(tenant_id, created_at)` indexing handles the Phase 1 volume (3000 rows/day is trivial for PG). Revisit if volume exceeds ~100K rows/day.

3. **Per-validator response versioning (immutable response chain).** Instead of storing `original_content` + `remediated_content` in one row, model each mutation as a separate row: `response_v0` (original), `response_v1` (after false-promise), `response_v2` (after identity-guard). This gives full audit trail of intermediate states. Current flat design is simpler and sufficient for Phase 1 (max 2 mutations: append + rewrite), but worth noting for future if the pipeline grows more stages.
