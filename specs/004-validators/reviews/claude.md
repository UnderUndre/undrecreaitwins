## Summary

Solid intent — porting business-safety guards (false-promise, format-injection, identity-and-provider-guard) into the engine before they're needed in court. The pipeline shape (input strip → generation → response validators) is correct and the active/dry-run mode design is the right rollout posture. The big problem isn't any single validator; it's the **combination of FR-015 ("all active by default") with the identity-guard's hardcoded Russian-language template containing a hardcoded persona name "Анна"** — on deploy day, every existing tenant whose personas are non-Russian or named anything else will start emitting broken-character replies on identity questions. That's a baseline-functionality break, not a polish issue. Several FR→task gaps round out the HIGH tier (tenant isolation has no task, 003-funnels composition has no task, error isolation has no task).

## Alternative approaches considered

- **Pre-generation prompt-engineering guard for false-promise (relates to F7)**: Inject a system-prompt hardening rule ("you are not authorized to offer discounts, delivery dates, refunds without operator approval — defer all such requests") instead of (or in addition to) the post-hoc judge. Cheaper because no judge call. Less reliable because LLMs ignore instructions under prompt pressure. Hybrid: prompt-engineer for primary defense, post-gen judge for backstop. Worth evaluating before defaulting to "every flagged reply costs an LLM call".
- **Batched judge across validators (relates to F7, drawn from context-for-review.md §2)**: Legacy `executeUnifiedValidation` batches validators sharing a model into one LLM call. Phase 1 only has one LLM-based validator (false-promise), so the batching surface is moot today — but Phase 2 (fact-grounding) will add another, and architecting for batching now is cheaper than retrofitting. Worth noting in plan as a forward-looking design constraint.
- **Discriminated-union `Verdict` type instead of single shape**: Current `Verdict.decision: VerdictDecision` is an open enum that doesn't constrain which other fields are populated when (e.g. `confidence` is required for judge-based verdicts, meaningless for deterministic strip). A discriminated union (`{ decision: 'pass' } | { decision: 'append_disclaimer'; disclaimerText: string; confidence: number } | …`) makes the type system enforce the shape, removing entire classes of "field missing" bugs. Worth weighing against the verbosity cost.
- **Per-tenant prefilter pattern table (relates to F8)**: Hardcoding patterns means a new false-promise pattern (a new product-line that "Yandex Plus" wasn't supposed to be promised, etc.) requires a code release. Patterns-in-DB allows ops to react without a deploy. Trade-off: more moving parts, more potential for tenant-specific drift. Phase 1 doesn't need this but Phase 2 might.
