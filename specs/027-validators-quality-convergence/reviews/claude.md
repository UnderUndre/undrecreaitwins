# SpecKit Review: 027-validators-quality-convergence

**Reviewer**: claude
**Reviewed at**: 2026-06-23T04:59:46Z
**Commit**: 0662b575a08fd1651488f1540eb3c4bb974fc006
**Artifacts reviewed**: spec.md, plan.md, tasks.md, data-model.md, contracts/quality-event-push.md, contracts/rules-reload.md, research.md, quickstart.md, constitution.md ŌĆö **cross-checked against live engine code** (`packages/core/src/...`)

## Summary

The *intent* is sound: two post-passes (`validateResponse` + DAR) genuinely duplicate machinery, and converging them at the engine instead of in product-028 UI is the right call. The **strategy** (build-on path B, tiered cheap-first, push-based unified log) is defensible.

But the **design was written against a fictional model of the existing code.** I read `validators/pipeline.ts`, `correction-rules/dar-pipeline.ts`, `correction-rules/types.ts`, `models/validators.ts`, and `chat-service.ts`. Almost every concrete API, column, and type the spec/plan/contracts/quickstart rely on is wrong. The quickstart's `runSystemValidator`/`runCustomRule` access fields (`result.verdict`, `result.rewritten`, `result.modifiedResponse`, `result.score`) that **do not exist** on the real return types ŌĆö proof the author never opened the files being "verified in code." The unified verdict mapping and the backfill `.sql` are keyed on columns (`passed`, `severity`, `runAt`, `validator_type`) that **do not exist** in `validator_runs`. The `UnifiedRule` model **drops `detector`**, without which DAR cannot run a single custom rule.

`analyze.md` returned PASS at 100% coverage ŌĆö but it mapped requirement-IDs to task-IDs without checking whether the tasks are *technically possible* against the codebase. Coverage ŌēĀ correctness. This is exactly the author-blind-spot Principle VI exists to catch.

**This is not ready for `/speckit.implement`.** None of the findings kill the *approach* ŌĆö they're all fixable by re-grounding the artifacts in the real signatures ŌĆö but they must be fixed and re-analyzed first.

## Alternative approaches considered

- **Adapter over refactor (lower-risk path B).** Rather than reshaping `ValidatorPipeline`/`darExecute` return types and the `QualityEventPush` wire format, introduce `ResponseGuard` as a *thin orchestrator* that calls the **real** `validateResponse(reply, ctx): Promise<string>` and the **real** `darExecute(llm, text, ctx): Promise<DARResult>` unchanged, then normalizes their existing outputs into the unified log at the emit boundary. This delivers FR-001 (single entry point), FR-004 (unified log) and FR-007 (path consistency) **without** touching the validator internals, the DAR internals, or the existing event contract ŌĆö which is what "build-on, not fork" (spec ┬¦Clarifications) actually implies. The current design claims build-on but in practice rewrites the type layer of both subsystems.
- **Unified log as a projection/view at the BFF, not a reshaped engine emit.** Keep the engine emitting its two native event shapes; do the coarse `verdict`+`detail` normalization in one BFF mapper that both 028 and audit read. Preserves `idempotencyKey`/`assistantId`/`snapshotVersion` end-to-end (F3) and keeps the engineŌåÆBFF contract stable. The spec rejected "UI normalization" ŌĆö but BFF-emitter-side normalization is a different, legitimate seam and avoids a breaking wire change.
- **Tier as configuration per call-site (resolves F7 cleanly).** Model the guard tiers (deterministic-only vs deterministic+LLM) as a per-call-site capability so agentic/buffered can opt into the cheap tier only, making cost/behavior parity a deliberate switch rather than an accident.
