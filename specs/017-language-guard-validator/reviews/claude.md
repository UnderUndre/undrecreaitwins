# SpecKit Review: 017-language-guard-validator

**Reviewer**: claude
**Reviewed at**: 2026-06-10T20:15:00Z
**Commit**: bd01847f4f3384f710e677240aa69360329bfc6c
**Artifacts reviewed**: spec.md, plan.md, tasks.md, research.md, checklists/requirements.md, .specify/memory/constitution.md

## Summary

The Language Response Guard spec is technically well-scoped and integrates cleanly into the existing validator pipeline. The deterministic Unicode approach is sound. However, the Gemini review raised a legitimate CRITICAL finding about code blocks in technical personas — this is real and the spec's own edge-case analysis contradicts itself on this point. Additionally, there are two HIGH findings: the non-compliant fraction formula is undefined (Gemini's F3 is correct), and the double DB query for the same config row is an architectural inefficiency that should be addressed. The `strip` remediation UX at high fractions is also questionable but not blocking.

## Findings

| ID | Severity | Area | Finding | Recommendation |
|---|---|---|---|---|
| F1 | CRITICAL | Logic / Edge Case | Gemini F1 is valid. Spec edge cases (line 81) say: "Code snippets contain ASCII operators and symbols from many scripts. The contamination fraction absorbs low-level mixed content; short inline code in a compliant response is unlikely to exceed `stripThreshold`." This assumption is false for technical personas. A Russian developer bot responding with a 30-line Python code block (~80% Latin/ASCII keywords) easily exceeds `blockThreshold: 0.30`. The response gets replaced entirely — the feature is broken for developer personas. Research.md §3 (line 116) even acknowledges "Code keywords in non-compliant script count toward fraction but are unlikely to exceed `stripThreshold` at 5%" — this hand-waves away the realistic scenario of large code blocks. | Exclude content inside markdown fenced code blocks (` ```...``` `) and inline code (`` `...` ``) from the non-compliant fraction calculation. This is deterministic — parse markdown fences before classification. The `ScriptClassifier.analyze()` method should receive the text with code spans zeroed out. Update spec.md Edge Cases, plan.md DD-001, and tasks.md T005 accordingly. |
| F2 | HIGH | Ambiguity / Spec | The formula for `nonCompliantFraction` is never explicitly defined. Spec says FR-003: "evaluate the response for content belonging to scripts outside `allowedLanguages` and produce a verdict" but doesn't specify the denominator. Is it `non_compliant_chars / total_chars`? Or `non_compliant_chars / (total_chars - common_chars)`? Or `non_compliant_chars / total_non_common_chars`? The choice fundamentally changes threshold behavior. Example: a response with 60% whitespace/punctuation, 30% Russian, 10% Chinese → with denominator `total_chars`, fraction = 10%. With denominator `classified_chars` (excluding Common), fraction = 25%. This makes SC-001 (≥95% compliant) and SC-005 (≥60% reduction) unmeasurable until the formula is pinned. | Add explicit formula to spec.md under Key Entities → LanguageGuardResult or as a new FR. Suggested: `nonCompliantFraction = nonCompliantScriptChars / (totalChars - commonChars)` where Common = punctuation, whitespace, digits, emoji, control chars. This matches the "meaningful script content" intuition. |
| F3 | HIGH | Performance / Architecture | Gemini F2 is valid. T008 (plan DD-003) adds a DB query for `validator_configs` inside `buildSystemPrompt()`. The validator pipeline's `resolveConfig()` already fetches the same row for the same `(tenantId, personaId, 'language-guard')` key. This is 2 DB reads for 1 config row per chat turn. For a high-throughput system this is measurable waste. | Resolve language-guard config once in the request lifecycle entry point (likely `ChatService.complete()` or equivalent) and pass the resolved config to both `buildSystemPrompt()` and the pipeline. This is a minor refactor — add an optional `preloadedConfigs` param or a request-scoped cache. |
| F4 | MEDIUM | UX / Logic | Gemini F4 has merit. `strip` remediation removes individual non-compliant characters. For a response that's 25% Chinese characters mixed with Russian (e.g., a bilingual explanation), stripping produces garbled text with missing characters mid-word. The spec says nothing about the UX quality of stripped output. At `stripThreshold: 0.05` (default), this is fine — a few characters removed. But a customer who sets `blockThreshold: 0.50` and `stripThreshold: 0.05` will see strips for 5-50% contamination, where upper-bound strips produce illegible text. | Document in plan.md DD-002: stripped output quality degrades significantly above ~15% contamination. Consider adding a `stripMaxFraction` cap (e.g., 0.15) beyond which the verdict is promoted to `block` regardless of `blockThreshold`. Alternatively, just warn operators in docs that high `blockThreshold` + strip = potentially ugly output. Not blocking for MVP but worth a follow-up. |
| F5 | MEDIUM | Types / Consistency | plan.md DD-001 maps `"ja" → [Hiragana, Katakana, Han, Latin]` and `"ko" → [Hangul, Han, Latin]`. Japanese includes Latin because of romaji and loanwords. Korean includes Han because of hanja. But `"zh" → [Han]` — Chinese only allows Han, not Latin. However, modern Chinese text routinely uses Latin characters for URLs, email addresses, and technical terms. A Chinese-only persona with `allowedLanguages: ["zh"]` will flag every URL and email as non-compliant. The `stripThreshold` (5%) absorbs small amounts, but a response containing a full URL (30+ Latin chars) in a 200-char Chinese message = 15% contamination → `strip`. This is the same class of issue as F1 but less severe. | Either add Latin to the `"zh"` script mapping (like `"ja"` and `"ko"` already do), or document this as a known limitation. Adding Latin is safer — modern Chinese text always contains some Latin. Same logic applies to `"ar"` (Arabic URLs) and `"he"` (Hebrew URLs). |
| F6 | MEDIUM | Hidden Assumption | spec.md FR-009 says "All language guard evaluation events — including `pass` — MUST be written to the validator audit log." But plan.md DD-005 says empty `allowedLanguages` → no audit entry. The pipeline needs to distinguish two different `pass` outcomes: (a) config exists, response clean → audit `pass`; (b) no config / empty config → no audit. The current `ResponseValidator` interface returns a `Verdict` — there's no "skip audit" signal. Tasks.md T003 adds a pipeline-level check for empty `allowedLanguages`, which is an implicit contract not captured in the interface. | Make the skip-audit signal explicit. Either: (1) return `null`/`undefined` verdict from `validateAndMutate` when config is empty (interface change), or (2) add a `shouldAudit` boolean to the return type, or (3) document the pipeline-level convention clearly in T003's description. Option (3) is lowest-cost for MVP. |
| F7 | LOW | Completeness | research.md §2.4 (line 106-108) says `'strip'` and `'pass'` are not in the `validatorVerdictEnum`. The plan only adds `'strip'` (T001). But `'pass'` is also missing from the enum. The spec (FR-009) requires auditing `pass` events. If `pass` isn't a valid enum value, the audit insert will fail at the DB level. | Clarify: if `'pass'` verdicts are stored as `'no_op'` (existing enum value) or as `NULL`, document this mapping in T001. If `pass` needs its own enum value, add it to the migration. |
| F8 | LOW | Completeness | plan.md DD-001 lists 10 script ranges. Unicode has many more blocks for Latin alone (Latin Extended-A/B/C/D, Latin-1 Supplement). The range `U+0041–U+024F` misses Latin Extended Additional (U+1E00–U+1EFF) which is used for Vietnamese and some African languages. If a tenant configures `allowedLanguages: ["vi"]`, Vietnamese diacritics would be classified as non-compliant. | For MVP, document the coverage limitation: Latin range covers Basic Latin + Latin-1 Supplement + Latin Extended-A/B. Vietnamese, Turkish dotted/dotless i, and other extended Latin may need additional ranges. Add a comment in `ScriptClassifier` noting the extensible design. |
| F9 | LOW | Stakeholder Clarity | spec.md SC-005: "The language directive reduces the raw violation rate by ≥60% compared to an identical persona without the directive." This is an empirical claim that can't be verified at design time. The 60% number is arbitrary. If the real reduction is 40%, does the feature fail its success criteria? | Flag SC-005 as "target, not gate" — same treatment as the requirements checklist already notes. Consider softening to "demonstrates measurable reduction" or keeping 60% but marking it as a stretch goal. |
| F10 | LOW | Terminology | spec.md uses "language" and "script" interchangeably but they're distinct concepts. Russian is a language, Cyrillic is a script. `"ru"` is a language code, but the guard checks scripts, not languages. A response in Ukrainian (language) using Cyrillic (script) passes `allowedLanguages: ["ru"]` even though it's not Russian. This is probably acceptable for the feature's purpose (preventing Chinese/Arabic contamination), but the terminology mismatch could confuse future maintainers. | Add a brief note in spec.md distinguishing the two: "This feature validates response scripts (Unicode code-point ranges), not natural language identification. `allowedLanguages` is a misnomer — it maps BCP-47 language codes to permitted Unicode scripts." |

## Alternative approaches considered

1. **Markdown-aware parsing**: Instead of excluding code blocks from the fraction, an alternative is to add a `codeAware: boolean` config flag that toggles code-block exclusion. This adds complexity but gives operators control. For MVP, always excluding code blocks is simpler and safer.

2. **Regex vs manual fence parsing**: For code-block exclusion, a simple regex (`/```[\s\S]*?```/g`) works for well-formed markdown but fails on nested or malformed fences. A state-machine parser is more robust. For MVP, regex is acceptable — the cost of a false negative (counting code as regular text) is just a higher fraction, not data loss.

3. **Script classification granularity**: Instead of broad ranges, one could use the Unicode `Script` property (via `Intl.v8BreakIterator` or a lookup table based on Unicode 15.0 script property data). This is more accurate but adds significant range table size. The static approach in DD-001 is sufficient for the 10 target scripts.
