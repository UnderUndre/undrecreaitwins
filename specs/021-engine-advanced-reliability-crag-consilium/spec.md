# Feature Specification: Engine Advanced Reliability — CRAG, Self-RAG, Consilium (Debate)

**Feature Branch**: `021-engine-advanced-reliability`
**Created**: 2026-06-18
**Status**: DRAFT
**Repo**: `undrecreaitwins` (Engine)
**Input**: `ai-agent-research-report.ru.md` (Industry patterns), `Архитектура-высоконадежных...md` (Scientific deep-dive).

## 1. Описание

Для промышленного использования в B2B продажах ИИ-агент должен обладать экстремальной надежностью. Стандартный RAG и одномодельная генерация склонны к галлюцинациям и "плывущему" тону в эмоциональных моментах. 

021 внедряет два продвинутых механизма надежности:
1.  **Corrective RAG (CRAG) & Self-RAG**: Многоступенчатая верификация найденных фактов перед их попаданием в генератор. Если фактов нет или они сомнительны — агент идет во внешний поиск или делает хандофф.
2.  **Consilium (Debate Pipeline)**: 3-агентный пайплайн (Генератор → Критик → Синтезатор) для обработки критических точек воронки (оплаты, возражения, конфликты).

## 2. Границы

**Это**: Рантайм-механики в `chat-service.ts` и `grounding-service.ts`. Оценка релевантности (Evaluator), семантическая маршрутизация (Router), 3-агентная оркестрация.

**Это НЕ**: UI настройки (Product 025). Базовый RAG (уже есть). Внешние инструменты поиска (используем абстракцию SearchTool).

## 3. Пользовательские истории

### US-1 — Вышибала фактов (CRAG Evaluator)
Когда бот извлекает документы из базы знаний (KB), отдельная быстрая модель (evaluator) оценивает каждый чанк на релевантность запросу юзера.
- **Correct**: Факты есть, идем в генерацию.
- **Ambiguous**: Мало инфы, идем в Web Search + KB.
- **Incorrect**: Инфы нет, идем строго в Web Search или "я не знаю".

### US-2 — Саморефлексия (Self-RAG)
При генерации ответа модель расставляет токены самопроверки. Если модель понимает, что утверждение не подтверждено документами (hallucination detected), ход автоматически перезапускается с другим контекстом.

### US-3 — Консилиум (Multi-Agent Debate)
Для сложных ходов (например, клиент злится или просит скидку 90%) включается пайплайн:
1. **Drafter**: Пишет черновик в стиле ассистента.
2. **Critic**: Проверяет черновик на огрехи (комплаенс, факты, тон).
3. **Synthesizer**: Выдает финальный ответ, исправляя ошибки критика.

## 4. Функциональные требования

### CRAG (Corrective RAG)
- **FR-001**: `KnowledgeEvaluator` модуль. Вход: Query + Chunks. Выход: Score [0-1] per chunk.
- **FR-002**: Порог уверенности (ConfThreshold). Если `max(scores) < 0.6` → статус `INCORRECT`.
- **FR-003**: При `INCORRECT` — автоматический вызов `TavilySearch` или `GoogleSearch` (через MCP/Tools).
- **FR-004**: Знаниевая очистка: удаление нерелевантных предложений из чанков перед подачей в LLM.

### Consilium (Debate)
- **FR-005**: Семантический роутер. Включает дебат-пайплайн только для `High-Stakes` ходов (настраиваемый список интентов или sentiment > 0.8).
- **FR-006**: Оркестратор пайплайна: последовательный вызов 3 ролей.
- **FR-007**: Поддержка разных моделей: например, Drafter = GPT-4o-mini, Critic = Claude 3.5 Sonnet (лучший критик), Synthesizer = GPT-4o.

## 5. Нефункциональные требования
- **NFR-1 (Latency)**: CRAG добавляет < 1.5 сек к ответу (при кэшировании оценок).
- **NFR-2 (Cost)**: Консилиум увеличивает стоимость хода в 3-4 раза. Должен быть выключен по умолчанию.

## 6. Success Criteria
- **SC-001**: Бот не выдумывает цену, если её нет в PDF, а честно говорит "надо уточнить" или ищет на сайте.
- **SC-002**: На агрессивное сообщение клиента ("вы мошенники!") отвечает идеально выверенный ответ от Синтезатора, прошедший цензуру Критика.
